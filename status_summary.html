Git‑Arbeitsfluss & Branching‑Strategie

Trunk‑Based (main) • PR erforderlich • Merge‑Commit • Releases via Tags • Optional: beta / release

Trunk & Pflicht

  • Trunk: main (Default)
  • PR Pflicht: keine direkten Pushes
  • Merge‑Strategie: Merge‑Commit (Standard)
  • Commits: feingranular, Conventional Commits

Arbeits‑Branches

  • feature/<thema> – neue Features
  • fix/<thema> – Bugs
  • chore/<thema> – Build/Doku
  • Namensschema: kebab-case

Release‑Strategie

  • Stable: Tags vX.Y.Z auf main
  • Pre‑Release: vX.Y.Z‑beta.N (optional)
  • Freeze: release/x.y.z nur bei Bedarf
  • Downloads über GitHub Releases (CI‑Artefakte)

Qualität & CI

  • PR‑Checks: cargo clippy -D warnings, yarn tsc --noEmit
  • Release‑Build: Tauri auf Tag v*
  • Optional: build auf PR, beta‑Builds
  • Docs: bei Architektur/Phase anpassen
1

main aktualisieren main

Was

Deinen lokalen main mit origin/main synchronisieren.

Warum

Verhindert Konflikte; neue Arbeit startet vom neuesten Stand.

git fetch --all --pruneRemote updaten, verwaiste Refs löschen
git switch mainauf main wechseln
git pull --ff-onlynur Fast‑Forward, kein lokaler Merge
2

Feature‑Branch anlegen feature/*

Was

Neuen Branch von main für die Änderung erstellen.

Warum

Saubere Historie, isolierte Arbeit, leichter PR.

git checkout -b feature/global-hotkeysBranch erstellen & wechseln
3

Arbeiten, committen, pushen

Was

Änderungen in kleinen Schritten committen und hochladen.

Warum

Feingranulare Historie; PR kann früh geprüft werden.

git add -pinteraktiv hunks auswählen
git commit -m "feat(hotkeys): add global hotkeys"klare Message (Conventional)
git push -u origin feature/global-hotkeyserster Push: Remote & Tracking setzen
git pushweitere Pushes genügen
4

Pull Request öffnen → main

Was

PR von feature/* nach main in GitHub öffnen.

Warum

Review, Kommentar‑Thread, CI‑Checks – alles an einem Ort.

Base: main • Compare: feature/global-hotkeysBeschreibung, Screens, Scope
5

CI‑Checks laufen lassen CI

Was

Automatisierte Qualitätssicherung auf PR.

Warum

Fehler früh erkennen; konsistente Qualität.

cargo clippy -D warningsRust Lint strikt
yarn tsc --noEmitTS Typecheck ohne Build
yarn buildoptional: Bundling test
6

Merge‑Commit durchführen

Was

„Create a merge commit“ auf der PR‑Seite.

Warum

Zwischen‑Commits bleiben erhalten; Historie nachvollziehbar.

(GitHub UI) Merge pull request → Create a merge commitStandard‑Strategie
7

Release taggen (CI baut Installer) vX.Y.Z

Was

Version markieren; GitHub Release mit Assets erzeugen.

Warum

Nutzer laden stabilen Installer direkt von der Releases‑Seite.

git tag v0.6.0 -m "Release v0.6.0"annotierter Tag
git push origin v0.6.0CI triggert Release‑Build
8

Optional: Beta / Release‑Freeze

Was

Vorabtests auf beta oder Freeze auf release/x.y.z.

Warum

Größere Releases stabilisieren; 1.0.x Patches ermöglichen.

Tag v0.6.0-beta.1Prerelease für Tester
git checkout -b release/1.0Freeze‑Branch, nur Bugfixes
